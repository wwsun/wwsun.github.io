# Write Blog Skill

撰写高质量博客文章。博客文章保存在 `content/blog/` 目录，风格专业、有深度，面向技术从业者。

## 工作流

### 第 1 步：访谈用户（如果信息不足）

如果用户只给了一个模糊的主题，先问以下问题（一次性全部问，不要分多轮）：

1. **核心观点是什么？** 这篇文章你最想传递的 1-2 个核心想法
2. **文章的读者是谁？** 面向开发者、产品经理、还是普通技术爱好者？
3. **是否有参考来源？** 比如一篇论文、一个工具、一个真实经历（有则填入 `source` 字段）
4. **大致结构？** 如果有提纲或关键小节，列出来

如果用户已经给了足够的信息（主题明确、有大纲、有内容），**跳过访谈，直接写**。

---

### 第 2 步：确定文件元数据

写作前先确定：

| 字段          | 规则                                                                               |
| ------------- | ---------------------------------------------------------------------------------- |
| `title`       | 中文标题，简洁有力，15字以内为佳                                                   |
| `date`        | 今天的日期，格式 `YYYY-MM-DD`                                                      |
| `tags`        | 1-4 个标签，kebab-case 格式（如 `ai-agent`、`product-thinking`）；优先复用已有标签 |
| `draft`       | 初稿设为 `false`；明显不完整时设 `true`                                            |
| `description` | 正文开头的 1-2 句核心概括，**禁止写"未命名"**，100-200 字以内                      |
| `source`      | 有参考来源则填 URL，原创则写 `null`                                                |

**文件名规则**：

- 小写英文 + 连字符（kebab-case）
- 简短描述性，2-5个单词，例如 `agent-skill-design.md`
- 避免中文字符（会影响 URL 编码）
- 保存路径：`content/blog/<filename>.md`

**常用标签参考**（优先复用）：
`ai`、`agent`、`claude`、`claude-code`、`llm`、`product-thinking`、`workflow`、`tools`、`learning`、`blog`、`skills`

---

### 第 3 步：撰写内容

#### 语言与语气

- **语言**：简体中文为主，技术术语可保留英文（如 LLM、Agent、Prompt）
- **语气**：专业、直接、有观点。避免空话套话
- **禁用句式**：
  - "在当今快速发展的..."
  - "本文将为您详细介绍..."
  - "总的来说..."（结尾收束句式）
  - 过度谦虚或过度卖弄的表达

#### 文章结构

根据文章类型选择合适结构：

**技术实践型**（工具使用、工作流）：

```
## 一、背景（为什么要做这件事）
## 二、核心做法（是什么、怎么做）
## 三、实际效果 / 案例
## 四、延伸思考（可选）
```

**思考观点型**（想法、框架）：

```
## 核心概念
## 分析与论证（2-3 个维度展开）
## 实践建议
```

**翻译整理型**（基于外部来源）：

```
> **原文标题**: ...
> **作者**: ...（如有）

TLDR（要点列表）

## 核心内容（按原文结构整理）
## 个人评注（可选）
```

#### Markdown 规范（Obsidian 格式）

**Callout 用法**：

```markdown
> [!note] 核心提示
> 这里放需要强调的关键信息

> [!warning] 注意事项
> 容易踩坑的地方

> [!info] 背景信息
> 补充性上下文
```

**Wikilink（内部链接）**：

- 引用其他笔记用 `[[笔记名]]`，不要用普通 Markdown 链接
- 例如：`[[Agent]]`、`[[Claude Code]]`、`[[OpenClaw]]`

**高亮**：用 `==文本==` 标记重要概念

**代码块**：指定语言，如 ` ```python `、` ```bash `、` ```markdown `

#### 内容质量标准

- **有具体案例**：抽象概念配实例，技术内容配代码
- **有个人观点**：不只是摘录信息，要有作者的判断和态度
- **结构清晰**：每个 H2 章节内容自洽，章节之间有逻辑递进
- **长度合适**：技术文章 800-2000 字；思考性文章 600-1500 字；翻译整理不限

---

### 第 4 步：创建文件

按如下模板创建文件，写入 `content/blog/<filename>.md`：

```markdown
---
title: 文章标题
date: YYYY-MM-DD
tags:
  - tag1
  - tag2
draft: false
description: >-
  这里写文章的核心摘要，100-200字，用于 SEO 和预览展示。
source: null
---

正文开始...
```

写完后输出文件路径，并用 1-2 句话说明文章结构。

---

## 约束

- **description 字段**：永远不能写"未命名"，必须是真实有意义的摘要
- **不创建重复文章**：写之前先快速确认 `content/blog/` 中是否已有相同主题的文章
- **不修改 quartz/ 目录**：只操作 `content/blog/` 下的文件
- **中文为主**：即使用户用英文问，文章默认用中文写（除非用户明确要求英文）
