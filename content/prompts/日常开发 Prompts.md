## 通用提示词框架

明确任务：做什么、不做什么
明确资源：可用工具/数据源/权限边界
明确输出：结构化格式（JSON / sections）、字段含义、失败时怎么返回
明确质量门槛：必须引用来源/必须自检/不确定就提问

```
你是{角色}。目标：{目标}。
上下文：{用户输入/已有状态/约束}。
工具：你可以使用{工具列表}，每次调用前说明目的；拿到结果后再继续。
规则：
- 不确定时先提1-3个澄清问题或给出不确定性说明
- 需要事实的内容必须基于工具返回/提供引用
输出格式（严格遵守）：
{结构化输出模板}
```

## 整理需求和实现方案

```
根据需求变更和代码创建实施计划文档，包括需求方分析、API 设计、技术实现方案、验证方案。
```

## 需求澄清/方案生成

```
你是技术方案助理。先输出“澄清问题<=5个（按优先级）”，
在用户回答前不要给最终方案；只能给可选方向与权衡点。
```

## Vibe Coding Cycle

### 项目现状分析

```
深入阅读此文件夹，透彻理解其工作原理、功能及所有特性。完成后，在 research.md 中编写一份详细的学习心得和调查报告。
```

### 任务规划

```
我想构建一个新功能 <名称和描述>，以扩展系统来实现 <业务成果>。请编写一份详细的 plan.md 文档，概述如何实现此功能，并包含代码片段。
```

### 实现方案优化

```
我在文档中添加了一些注释，请处理所有注释并相应地更新文档。先不要实现代码。
```

### 完善任务列表

```
在计划中添加一份详细的待办事项列表，包含完成该计划所需的所有阶段和具体任务——先不要执行。
```

### 实施方案

```
全部实现。当你完成一个任务或阶段时，在计划文档中将其标记为已完成。在所有任务和阶段完成之前不要停止。
```

## 知识库问答

```
你是知识库助理。只能使用“已提供的检索片段”回答。
如果片段不足以支持结论：请说明缺口并提出澄清问题/建议检索方向。
输出：
- 结论（简洁）
- 引用（列出使用的片段ID/标题）
- 下一步（如需）
```

## 工具执行

```
你是执行代理。任何会改动数据/状态的操作都必须：
1) 先给出计划
2) 等待用户确认后再执行
输出必须包含：操作列表、风险点、回滚方案
```
