## 阅读理解

拒绝“稍后读”变成“从不读”。在信息进入 Obsidian 之前，先用 LLM 进行降噪。

```
阅读这篇文章，为我提取 3 个核心技术洞见（Key Insights）和 1 个我作为开发者可以直接尝试的具体行动（Actionable Item）。请忽略通用的背景介绍。输出格式为 Markdown，包含原文链接。
```

## 学习新的领域

### 导师模式

```
我是一个初学者，想要深入理解 `[填入你的陌生领域]`。 请你扮演一位严厉但循循善诱的苏格拉底式导师。**请不要直接给我讲课或列大纲。**

请先针对这个领域最核心的概念，向我提出 1 个具有挑战性的问题。等待我的回答。 无论我回答得如何，请基于我的答案进行反馈：指出我的盲区，修正我的误解，并引导我进行更深一层的思考。如果你准备好了，请抛出第一个问题
```

不让 AI 直接“倾倒”信息（那样只会增加你的认知噪声），而是先通过提问来测定你的知识边界，确立坐标系

### 类比

```
关于我们刚才讨论的 `[刚才提到的具体概念]`，请用一个生活中的通俗类比来解释它。
解释完类比后，请立刻告诉我，这个类比在哪些方面是**不准确**或**甚至错误**的？真实的技术/理论细节在哪些地方比这个类比更复杂？ 我需要知道‘简化的代价’是什么。
```

当你对核心概念有模糊认知后，需要通过类比（Model）来压缩信息，但同时要警惕类比带来的失真。

### 反直觉

```
大多数关于 `[填入你的陌生领域]` 的入门介绍中，有哪些‘常识’其实是过时的，或者是被过度简化的？ 请给我提供一个该领域内**反直觉**的、或者**极具争议性**的观点。为什么这个观点虽然冷门，却对理解本质至关重要？
```

避免陷入“回声室”效应。LLM 倾向于给出平庸、正确的共识性回答，你需要逼迫它吐出具有“高惊奇度”（Surprisal）的信息。
